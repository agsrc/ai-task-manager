# Task Architect: AI-Agentic Microservice

- An intelligent task management system built with Java 25 and Quarkus 3.34. This project demonstrates how to transform unstructured "brain dumps" into strictly typed database entities using Local LLMs (Ollama) and LangChain4j.

## 🚀 Key Features
- Natural Language Extraction: Uses Phi-4-mini via Ollama to parse messy text into structured tasks.

- Temporal Intelligence: Automatically maps relative terms like "tomorrow" or "next week" into LocalDateTime timestamps.

- Virtual Threaded AI I/O: Utilizes Project Loom (@RunOnVirtualThread) to handle blocking AI calls without exhausting OS threads.

- Clean Architecture: Strict separation between AI Data Transfer Objects (DTOs), API Records, and JPA Entities.

- Local Privacy: 100% local execution—no data leaves your machine.

## 🛠️ Tech Stack
- Runtime: Java 25 (LTS)

- Framework: Quarkus 3.34.1 (Supersonic Subatomic Java)

- AI Engine: Ollama running phi-4-mini:3.8b

- Integration: Quarkus LangChain4j (Anthropic-ready)

- Database: H2 (File-based persistence) / Hibernate Panache

## 🛰️ The AI Workflow: From "Brain Dump" to Database

**When architect.parse(prompt),not just a method is called; a sophisticated proxy chain is triggered.**

### The Sequence:

- The Proxy Trigger: Since TaskArchitect is an interface, Quarkus creates a Proxy Class at startup. When you call parse(), this proxy intercepts the string.

- Prompt Engineering (Hidden): LangChain4j combines your @SystemMessage, your @UserMessage, and a JSON Schema it generated by reflecting on your TaskListWrapper class.

- The HTTP Handshake: Quarkus sends a REST POST request to localhost:11434 (Ollama).

- Inference (The 20-45s Wait): Ollama loads the phi-4-mini weights into RAM. The model predicts tokens one by one until it completes the JSON block.

- Output Parsing: The response comes back as a raw string. Quarkus uses Jackson to map that string into your TaskListWrapper object.

- Virtual Thread Yielding: Because of @RunOnVirtualThread, the carrier thread was released to do other work while waiting for Ollama, making this highly efficient.

## 🏗️ Architectural Decisions
1. The Wrapper Pattern
   To ensure robust JSON parsing from the LLM, we implemented the Wrapper Pattern. Instead of returning a raw JSON array, the AI returns a TaskListWrapper POJO. This avoids "Type Erasure" issues during the Quarkus augmentation phase and provides a stable schema for the reflection-based parser.

2. DTO vs. Entity Separation
   AiTaskDTO: A lightweight class optimized for AI reflection and extraction.

- TaskDTO (Record): An immutable Java Record used for API responses (includes IDs and completion status).

- Task (Entity): The JPA model used for database persistence.


## 🚦 Getting Started
Prerequisites
Ollama: [Download here](https://ollama.com/)

Model: Run ollama run phi4-mini in your terminal.

Configuration
Update your src/main/resources/application.properties:

Properties

`quarkus.langchain4j.ollama.chat-model.model-id=phi4-mini`

`quarkus.datasource.jdbc.url=jdbc:h2:file:./data/tasksdb;DB_CLOSE_DELAY=-1`

`quarkus.hibernate-orm.database.generation=update`

### Running the App
- Bash
`./mvnw quarkus:dev`
- 🧪 Sample Usage
Endpoint: POST `/tasks/ai`

Input: > `"I need to fix the JPA bug by tomorrow and meet with  a friend on Friday."`

Result:
The system extracts two tasks, assigns priorities (1-5), and calculates the dueDate based on the current server time.
